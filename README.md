# Innotech Workspace Kanban

Учебная канбан-доска на русском языке: регистрация, вход, рабочие пространства, добавление участников по логину, проекты, задачи, чеклисты, загрузка файлов, материалы, комментарии, история и отложенные задачи.

## Запуск

```bash
docker compose up -d db
npm run dev
```

По умолчанию backend подключается к PostgreSQL `postgres://kanban:kanban@127.0.0.1:55432/kanban` и открывается на `http://localhost:5173`. Если порт занят:

```bash
PORT=5174 npm run dev
```

Для аварийного локального запуска без PostgreSQL можно использовать старое JSON-хранилище:

```bash
npm run dev:json
```

## Проверки

```bash
npm run build
```

Команда выполняет синтаксическую проверку JavaScript и тесты бизнес-логики.

## Что показывать на защите

1. Зарегистрировать пользователя. После регистрации автоматически создаются `Мой workspace` и проект `Канбан`.
2. Создать второго пользователя, вернуться владельцем и добавить его в workspace по логину.
3. Создать задачу с исполнителем, приоритетом, сроком, тегом, чеклистом и материалом.
4. Прикрепить файл к задаче и показать, что он сохраняется как материал.
5. Открыть задачу справа, добавить комментарий и показать историю действий.
6. Попробовать закрыть задачу с невыполненным чеклистом: система не даст закрыть ее без готовности.
7. Выполнить чеклист, переместить задачу в `Готово`.
8. Отложить задачу с причиной, открыть режим `Отложено`, затем вернуть задачу на доску.

## Зачем эти функции

Функции выбраны не как абстрактная автоматизация, а как конкретная поддержка командной работы:

- авторизация и workspace дают приватное пространство команды;
- добавление по логину проще для учебного проекта, чем приглашения по почте;
- чеклист фиксирует критерии готовности и снижает риск закрыть сырую задачу;
- загрузка файлов привязывает ТЗ, макеты и отчеты прямо к задаче;
- комментарии и история сохраняют контекст решений;
- отложенные задачи убирают заблокированную работу с активной доски, но не теряют ее;
- проекты помогают разделять несколько потоков внутри одного workspace.

## Хранение данных

Основные данные лежат в PostgreSQL. Загруженные файлы сохраняются локально в `uploads/` и игнорируются git. JSON-хранилище осталось только как fallback для локальной проверки через `npm run dev:json`.
